Update: direct comparison of Python working path vs MIDAS frontend path (2026-03-27)
- Compared API sequences in:
  - `~/Documents/wc_event_rate_workaround.py`
  - `~/Documents/wc_capture_waveforms_png.py`

  - `OpenDevice -> ResetDevice -> SetDefaultParameters -> SetChannelState -> SetTriggerSourceState -> SetTriggerEdge -> SetTriggerThreshold -> SetTriggerMode -> PrepareEvent -> AllocateEventStructure -> StartRun -> ReadEventBuffer -> DecodeEvent -> ReadChannelDataStruct -> StopRun`

Key divergence identified (root cause for missing WCWF payloads in some MIDAS runs):
- MIDAS frontend previously dropped waveform payloads when all samples were non-finite:
  - it logged `Skipping ch=... waveform with 0/... finite samples` and did not write `WCWF`.
  - This can produce runs with non-zero `Events sent` but `WCWF Length: 0`.
- Python scripts did not apply that drop rule for MIDAS banking (they keep processing from direct channel data).

MIDAS-side fix implemented:
- In `wc_midas_frontend.cxx`, changed waveform handling to store raw waveform payload whenever `WaveformData` pointer is valid and `WaveformDataSize > 0`, even if finite-count is zero.
- Added explicit counters for:
  - null waveform pointer (`g_wave_ptr_null`)
  - zero waveform size (`g_wave_size_zero`)
  - non-finite-only waveform (`g_wave_nan_only`)

Secondary divergence fixed:
- Software-trigger pacing in MIDAS poll loop was effectively too aggressive (`SendSoftwareTrigger()` in tight loop + 1 ms sleep), unlike Python scripts which schedule by requested trigger period.
- Updated MIDAS to period-based software trigger scheduling using `ss_millitime()` and `g_sw_trigger_hz`.

Remaining blocker (orthogonal to API sequence):
- Shared-host MIDAS RPC/frontend instability (start aborts/timeouts, occasional exit 137) intermittently prevents clean run transitions and can still produce empty/truncated runs.

Update: MIDAS BOR transition diagnosis and mitigation (2026-04-01)
- User-visible symptom:
  - MIDAS stack starts, but START can fail with:
    - `previous transition did not finish yet`
    - `cm_transition status 603` from WaveCatcher frontend
  - Runs then do not record events.

Confirmed root cause:
- `WAVECAT64CH_OpenDevice()` can block unpredictably inside FTDI/libusb internals.
- When this call happens in BOR transition context (`begin_of_run` / `rc_transition`),
  MIDAS transition RPC can stall and leave apparent stuck transition state.

Implemented fix in native frontend (`midas_frontend/wc_midas_frontend.cxx`):
1) Remove device open from BOR path:
   - BOR no longer calls `OpenDevice`.
2) Async open worker at frontend startup:
   - device open now runs outside transition callback.
3) Explicit readiness state exposed in ODB:
   - `/Equipment/WaveCatcher/Variables/device_open_state`
   - `/Equipment/WaveCatcher/Variables/device_open_state_str`
   - States: `0 idle, 1 in_progress, 2 ready, 3 failed, 4 timed_out`
4) Open timeout guard:
   - `in_progress` promoted to `timed_out` after 180 s.
5) BOR behavior:
   - fast-fail with explicit message when device not ready, instead of hanging transition.
6) Runtime stability patch:
   - throttled readiness ODB publishing + idle sleep to avoid frontend CPU spin while idle.

Operational mitigations in CLI/start scripts:
- `scripts/wc_run_midas_cli.sh`:
  - added `--start-retries N`
  - waits for `device_open_state==2` before START
  - skips START when hardware is not ready (avoids transition churn)
  - clears stale `/Runinfo/Transition in progress`
  - can restart frontend between retries to retrigger async open
- `scripts/wc_start_midas_stack.sh`:
  - staged timeout profile and hardware preflight retained
  - stale transition lock clear retained

Interpretation:
- This converts failure mode from transition deadlock to bounded readiness gating.
- If hardware open is slow/fails, operators get explicit readiness status and retry path
  without wedging MIDAS transition control flow.
