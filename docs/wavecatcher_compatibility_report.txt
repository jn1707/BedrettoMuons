Update: direct comparison of Python working path vs MIDAS frontend path (2026-03-27)
- Compared API sequences in:
  - `~/Documents/wc_event_rate_workaround.py`
  - `~/Documents/wc_capture_waveforms_png.py`

  - `OpenDevice -> ResetDevice -> SetDefaultParameters -> SetChannelState -> SetTriggerSourceState -> SetTriggerEdge -> SetTriggerThreshold -> SetTriggerMode -> PrepareEvent -> AllocateEventStructure -> StartRun -> ReadEventBuffer -> DecodeEvent -> ReadChannelDataStruct -> StopRun`

Key divergence identified (root cause for missing WCWF payloads in some MIDAS runs):
- MIDAS frontend previously dropped waveform payloads when all samples were non-finite:
  - it logged `Skipping ch=... waveform with 0/... finite samples` and did not write `WCWF`.
  - This can produce runs with non-zero `Events sent` but `WCWF Length: 0`.
- Python scripts did not apply that drop rule for MIDAS banking (they keep processing from direct channel data).

MIDAS-side fix implemented:
- In `wc_midas_frontend.cxx`, changed waveform handling to store raw waveform payload whenever `WaveformData` pointer is valid and `WaveformDataSize > 0`, even if finite-count is zero.
- Added explicit counters for:
  - null waveform pointer (`g_wave_ptr_null`)
  - zero waveform size (`g_wave_size_zero`)
  - non-finite-only waveform (`g_wave_nan_only`)

Secondary divergence fixed:
- Software-trigger pacing in MIDAS poll loop was effectively too aggressive (`SendSoftwareTrigger()` in tight loop + 1 ms sleep), unlike Python scripts which schedule by requested trigger period.
- Updated MIDAS to period-based software trigger scheduling using `ss_millitime()` and `g_sw_trigger_hz`.

Remaining blocker (orthogonal to API sequence):
- Shared-host MIDAS RPC/frontend instability (start aborts/timeouts, occasional exit 137) intermittently prevents clean run transitions and can still produce empty/truncated runs.

- Result: PyDAQ can often recover/continue where MIDAS BOR semantics require bounded completion.

Update: Python-first failure learning and MIDAS race fix (2026-04-10)

Why we forced Python DAQ to fail first:
- We reproduced Python instability by adding intrusive "INFO/snapshot" calls around critical API calls
  (extra device queries like `GetLibraryVersion`, `GetWaveCatDevices`, `GetDeviceInfo` interleaved
  with acquisition setup).
- In that instrumented state, Python often shifted failures to config calls (`ResetDevice` / `SetDefaultParameters`)
  and produced no events.
- After removing intrusive snapshot polling and keeping only lightweight call/result logs,
  Python recovered and again produced non-zero decoded events.

Transfer of this learning to MIDAS:
- Treat WaveCatcher setup calls as an uninterrupted critical section.
- Do not inject extra device-query API calls during open/config transitions.

Threading/race root cause in MIDAS frontend:
- In `wc_midas_frontend.cxx`, device open was performed in an async worker.
- The bug: `g_device_open=true` was set immediately after `OpenDevice` succeeded, before
  `ResetDevice` + `SetDefaultParameters` completed in that same worker.
- BOR checked `g_device_open` and could proceed into `wc_apply_run_configuration()` while the open worker
  was still executing reset/default, creating overlapping WaveCatcher API activity across paths.
- This is the interrupt/race condition that matched the Python "intrusive interruption" pattern.

Implemented fix:
1) Readiness publication fixed:
   - `g_device_open` is now set only after full `OpenDevice -> ResetDevice -> SetDefaultParameters` success.
   - BOR gates on ready state (`device_open_state==2`) plus `g_device_open`.
2) Open sequencing made deterministic in init path:
   - open worker runs as a single startup sequence from `frontend_init` (no detached overlap with BOR).
3) Additional stability fix discovered during validation:
   - BOR crashed in MIDAS `message_print` due an oversized single log line; split into shorter log lines.

Validation after fix (same target settings as Python check):
- Trigger: normal, positive edge, threshold `0.020 V`, channel `0`, duration `10 s`.
- MIDAS run starts and stops cleanly (no transition timeout/RPC abort).
- Non-zero acquisition confirmed: `events_sent/decode_hits = 914` in run `#2333`.
